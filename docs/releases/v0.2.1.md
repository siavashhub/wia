# WIA v0.2.1

> Replaces the withdrawn **v0.2.0** release. v0.2.0 shipped with a packaging
> bug that caused the installed app footer to display the wrong version
> (`v0.1.0`). v0.2.1 contains every v0.2.0 feature plus the fix.

This is the second public release of WIA and the first release with the new
single-source-of-truth versioning pipeline. The git tag is now the only thing
you bump — `release.yml` stamps `version.json` and the installer
automatically.

## Highlights

- **Smarter categorization.** A category-umbrella model groups related signals
  (calendar / Teams / email) under a single bucket and learns from the data
  scanned for the week. Multiple iterations across the v0.2 cycle improve
  precision and reduce duplicates.
- **Editable, mergeable entries.** Add manual entries, take per-entry notes,
  tag signals inline, change durations, and delete by category group.
- **High-impact filtering.** Surface the work that mattered most via the new
  impact/influence model and a "high impact" filter.
- **Grouped filters & private-event filtering.** Filter the briefing view by
  category, and hide private calendar items.
- **Organization auto-detect.** Inferred from the signed-in identity so you
  don't have to configure it.
- **ISO 8601 weeks.** Briefing weeks now follow the ISO standard; week-start
  preference lives in Preferences instead of the toolbar (#15).
- **Daily-rotating logs.** New `logging_setup` writes to
  `%LOCALAPPDATA%\WIA\WIA\logs\wia.log`, 30-day retention, configurable via
  `WIA_LOG_*` env vars. User content stays at DEBUG.

## Features and improvements

### Categorization & grouping
- Category umbrella model that aggregates related signals into one entry.
- Scan-item merge & deduplication so a meeting joined from multiple sources
  appears once.
- Multiple categorization-quality passes during the release cycle.
- Orchestrator now carries additional metadata to feed the new category model.

### Briefing UI
- Add manual entries directly from the UI.
- Per-entry notes.
- Inline signal tags.
- Edit duration on existing entries.
- Sort table columns; improved table layout/spacing.
- Group-based filters; toggle for private events.
- General visual polish across multiple iterations.

### Preferences
- Week-start day moved from the toolbar into Preferences.
- Default week handling switched to ISO 8601 (#15).

### Identity / Work IQ
- Auto-detect organization from the Work IQ CLI identity.
- Calendar merge improvements and retry handling around the Work IQ MCP
  client.

### Logging
- New rotating file logger (`logging_setup.py`), default INFO, daily rotation,
  configurable retention.
- `GET /api/health/logs` exposes the active log file path, level, and
  retention for self-diagnosis.

## Fixes

- **Packaging:** bundle `version.json` inside the frozen exe so the installed
  app reports the correct version in the footer and `/api/health` (#18). This
  is the bug that withdrew v0.2.0.
- **Week anchor:** corrected an off-by-one in week-boundary calculation.
- **Tags:** assorted tag-handling regressions.
- **Duplicates:** orchestrator no longer surfaces duplicate entries across
  overlapping signals.

## Release pipeline

- **Single source of truth for version (#17).** Push a `vX.Y.Z` git tag and
  `release.yml` stamps `version.json`, runs PyInstaller, drives Inno Setup
  with `MyAppVersion=X.Y.Z`, and publishes the release. No more bumping
  multiple files by hand.
- Release notes are now sourced from `docs/releases/<tag>.md` when present
  and appended with GitHub's auto-generated "What's Changed" section.

## Upgrade notes

- Installing v0.2.1 over v0.2.0 / v0.1.0 is supported in place; user data at
  `%LOCALAPPDATA%\WIA\WIA\` is preserved.
- No configuration changes required.
- The internal `version.json` shipped in source now reads `0.0.0-dev` — this
  is intentional and only affects developers running `uv run wia-desktop`
  from a source checkout.

## Verifying the download

Each release ships `SHA256SUMS.txt`. Validate on Windows with:

```pwsh
Get-FileHash .\wia-setup-0.2.1.exe -Algorithm SHA256
```

and compare against the value in `SHA256SUMS.txt`. Build provenance
attestations are also published; see [docs/AUTH.md](../AUTH.md) and the
release assets list.
